# Behavioral Corrections DB — Design Spec

**Date**: 2026-03-25
**Status**: Approved
**Scope**: All agents (chief-builder, dev, future agents)

---

## Problem

Agent behavior is currently static — defined by behavior files and persona files committed to the repo. When a user corrects an agent's output on a specific ticket, that learning disappears after the session. There is no mechanism to:

- Persist corrections across sessions
- Scope corrections to a specific project vs globally
- Progressively promote validated corrections into the core behavior files
- Activate or deactivate specific corrections without touching source files

---

## Design Overview

A two-layer SQLite database system that stores behavioral corrections with a lifecycle (`active → pending_integration → integrated | inactive`), loaded by agents at startup as dynamic constraints layered on top of static behavior files.

```
GitHub comment @cao-learn
        │
        ▼
   lib/corrections.py  ──► ~/.claude/projects/<slug>/cao.db  (project-pattern)
        │               ──► ~/.claude/cao.db                  (general)
        │
        ▼
   Agent startup (step 0)
   └── load active corrections → inject as constraints into deliberation
        │
        ▼
   /cao-corrections promote <id>
   └── agent analyzes target file + conflicts → proposes diff → user validates → commit
        │
        ▼
   status: integrated — absorbed into file, no longer loaded at runtime
```

## Storage

### Two SQLite files — same schema

| File | Scope | Contains |
|------|-------|---------|
| `~/.claude/cao.db` | Global — all projects | `class: general` corrections |
| `~/.claude/projects/<slug>/cao.db` | Per project | `class: project-pattern` corrections |

`<slug>` = current working directory with `/` replaced by `-` (matches existing CAO memory convention).

**Directory creation**: `lib/corrections.py` creates the directory and bootstraps the schema on every call if either does not exist. This is idempotent — safe to call multiple times. No external setup required; the agent startup `load` call is sufficient to initialize both DBs on first use.

### Schema

```sql
CREATE TABLE corrections (
  id              TEXT PRIMARY KEY,
  -- Format: <prefix>_<slug>_<ticket>_<keyword> — see ID Generation below
  agent           TEXT NOT NULL,
  -- "chief-builder" | "dev" | "*"
  -- "*" = applies to all agents, maps to agents/behaviors/ on promotion
  class           TEXT NOT NULL CHECK (class IN ('project-pattern', 'general')),
  gap             TEXT NOT NULL,
  -- what was missing or wrong in the original behavior
  rule            TEXT NOT NULL,
  -- 1-sentence rule to apply in future sessions
  source          TEXT,
  -- GitHub reference, ex: "pascalpldev/instavid#7"
  source_comment_id TEXT,
  -- GitHub comment ID — used for deduplication across agent re-runs
  status          TEXT NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'inactive', 'pending_integration', 'integrated')),
  target_hint     TEXT,
  -- inferred promotion target, ex: "agents/behaviors/yagni.md"
  integrated_commit TEXT,
  -- git commit SHA once integrated
  integrated_file TEXT,
  -- file path modified during integration
  created_at      TEXT NOT NULL,
  updated_at      TEXT NOT NULL
);
```

Schema is bootstrapped by `lib/corrections.py` directly (no separate migration file needed — single `CREATE TABLE IF NOT EXISTS` on init).

### ID Generation

IDs are generated by `lib/corrections.py` at insert time using this algorithm:

```
prefix     = agent abbreviation: "cb" (chief-builder) | "dev" | "all" (* agent)
project    = last segment of PROJECT_SLUG, max 12 chars, alphanumeric only
ticket     = ticket number if from source, else "manual"
keyword    = first meaningful word from `rule`, lowercased, alphanumeric only, max 12 chars

id = f"{prefix}_{project}_{ticket}_{keyword}"
```

Examples:
- `cb_instavid_7_ratelimit` — chief-builder, instavid project, ticket #7, rule about rate limits
- `all_global_manual_firstuser` — all agents, global, manual entry, rule about first user
- `dev_myapp_12_migration` — dev agent, myapp project, ticket #12, rule about migrations

**Collision handling**: if an ID already exists, append `_2`, `_3`, etc.

## Correction Creation

### Path 1 — GitHub comment `@cao-learn` (primary)

Written by the user in any GitHub issue comment during a ticket review:

```
@cao-learn
gap: rate limits were not analyzed before proposing the endpoint
rule: always check existing instavid API rate limits before any endpoint design
agent: chief-builder
```

Short form (agent inferred from ticket context):

```
@cao-learn gap="X" rule="Y"
```

**Agent behavior on detection:**
1. Extracts `gap`, `rule`, `agent` (inferred if absent), `class` (inferred from rule specificity)
2. Checks `source_comment_id` for deduplication — skips silently if already saved
3. Inserts into appropriate DB with `status: active` (project vs global based on `class`)
4. Cross-checks the new rule against loaded core behavior files (agent reads them semantically)
5. If conflict detected with a core rule → `corrections.py update <id> --status inactive`, post conflict alert (see below)
6. If no conflict → post confirmation comment:

```
Correction saved [#cb_instavid_7_ratelimit]
Rule: "always check instavid API rate limits before any endpoint design"
Agent: chief-builder | Scope: project-pattern | Status: active
To promote to core: /cao-corrections promote cb_instavid_7_ratelimit
```

**Conflict alert comment (step 5):**

```
[cao-corrections] Correction #cb_instavid_7_ratelimit saved but set to INACTIVE.
Conflict detected with core rule in agents/behaviors/yagni.md:
  Core rule:  "<extracted rule from the file>"
  Correction: "<rule from @cao-learn>"

To override the core rule and activate: /cao-corrections activate cb_instavid_7_ratelimit
To integrate and resolve the conflict:  /cao-corrections promote cb_instavid_7_ratelimit
```

The conflict check is performed by the **agent** (not by `corrections.py`) because detecting semantic contradiction requires reading and understanding behavior file content. `corrections.py` only exposes the `update --status` command used to deactivate the correction.

**`agent` inference rules:**
- Rule references project-specific details (service names, stack, conventions) → `agent` = triggering agent, `class` = `project-pattern`
- Rule is universal methodology → `agent` = triggering agent, `class` = `general`
- Rule applies to all agents equally → `agent` = `*`, `class` = `general`

**`class` / `agent` relationship and promotion targets:**

| class | agent | Promoted to |
|-------|-------|-------------|
| `general` | `*` | `agents/behaviors/<file>.md` (shared core) |
| `general` | `chief-builder` | `agents/positions/chief-builder/agent.md` or persona |
| `general` | `dev` | `agents/positions/chief-builder/personas/dev.md` |
| `project-pattern` | `chief-builder` | `agents/positions/chief-builder/agent.md` |
| `project-pattern` | `dev` | `agents/positions/chief-builder/personas/dev.md` |

**Note**: `project-pattern` + `agent: *` is **not a valid combination**. A rule that applies to all agents is by definition not project-specific — it belongs in `agents/behaviors/` (shared core). If the user writes `agent: *` with a project-scoped rule, the agent reclassifies it as `general` and notes the correction in the confirmation comment.

### Path 2 — CLI (manual entry, outside ticket context)

```bash
/cao-corrections add \
  --agent chief-builder \
  --rule "always ask who is the first user of a feature (internal or external)" \
  --gap "user type not considered during scope definition" \
  --class general
```

## Lifecycle

```
active ──────────────────────────────► inactive
  │                                    (ignored by agents, kept for history)
  │
  ▼
pending_integration
  │  (agent has analyzed target file + proposed diff, awaiting user validation)
  ▼
integrated
  (rule absorbed into a source file via commit — no longer loaded at runtime)
```

---

## CLI Skills

```bash
# List
/cao-corrections list                     # active corrections (project + global)
/cao-corrections list --all               # all statuses
/cao-corrections list --agent dev         # filter by agent
/cao-corrections list --status pending_integration

# Manage
/cao-corrections deactivate <id>          # active → inactive
/cao-corrections activate <id>            # inactive → active
/cao-corrections status <id>              # full detail for one correction

# Promote to core
/cao-corrections promote <id>             # triggers analysis + diff proposal
```

### `/cao-corrections promote <id>` — detailed behavior

1. Load correction (`gap`, `rule`, `agent`, `class`, `target_hint`)
2. Resolve promotion target:
   - If `target_hint` is set and file exists → use it
   - Else if `target_hint` is set but file does not exist → log warning, fall back to `class`+`agent` table
   - Else → use `class`+`agent` table to select target file; if multiple candidates, present list and ask user to pick
3. Read target file in full
4. Check for: conflicts with existing rules, redundancy, contradictions
5. Propose exact diff (same interactive model as a brainstorming session)
6. On user validation: write file, commit, update `status → integrated`, set `integrated_commit = <sha>` and `integrated_file = <path>`
7. No automatic write — user validates before any file change

## Agent Loading at Startup

Added to **step 0 (Init)** of every agent:

```bash
PROJECT_SLUG=$(pwd | tr '/' '-')

CORRECTIONS=$(python3 lib/corrections.py load \
  --agent "$AGENT_NAME" \
  --project-db "$HOME/.claude/projects/${PROJECT_SLUG}/cao.db" \
  --global-db  "$HOME/.claude/cao.db")
# Exit code 0 always (empty output if no corrections — not an error)
```

Output injected into agent context as a constraints block (empty string if no active corrections — no block added):

```
## Active corrections (loaded at startup)

[project-pattern — #cb_instavid_7_ratelimit | chief-builder]
Gap: rate limits not analyzed before endpoint design
Rule: always check instavid API rate limits before any endpoint design

[general — #all_global_manual_firstuser | *]
Rule: ask who is the first user of a feature (internal or external) before proposing scope
```

**Precedence**: corrections take priority over static behavior files when they conflict — they represent explicit human validation of a refinement to the base policy. The agent flags the override in its `## Adaptations` section.

## `lib/corrections.py` — New library

New file added alongside `lib/logger.py`. Two interfaces:

**CLI subcommands:**

| Subcommand | Parameters | Output |
|------------|-----------|--------|
| `load` | `--agent <name>` `--project-db <path>` `--global-db <path>` | Formatted constraints block (stdout) or empty string. Exit 0 always. |
| `add` | `--agent <name>` `--rule "..."` `--gap "..."` `--class <class>` `--db <path>` `[--source "..."]` | `id` of created correction (stdout). Exit 0 on success, 1 on error. |
| `parse-and-save` | `--comments <json>` `--agent <name>` `--source <ref>` `--project-db <path>` `--global-db <path>` | One line per saved correction: `SAVED <id>` or `SKIPPED <comment_id> already_exists`. Exit 0 always. |
| `update` | `<id>` `--status <status>` `--db <path>` | Exit 0 on success, 1 if id not found. |
| `get` | `<id>` `--db <path>` | JSON of correction. Exit 0 on success, 1 if not found. |
| `list` | `--agent <name>` `--status <status>` `--db <path>` | JSON array. Exit 0 always (empty array if none). |

**`parse-and-save` specification:**

Input: JSON array of GitHub comment objects (from `gh issue view --json comments`). Each comment has `.databaseId` (used as `source_comment_id`) and `.body`.

For each comment containing `@cao-learn`:
1. Parse block format (multiline `key: value`) or short form (`key="value"`)
2. Extract: `gap` (optional), `rule` (required — skip comment with warning if absent), `agent` (optional), `class` (optional)
3. Infer missing `agent` from `--agent` parameter (the running agent's name)
4. Infer missing `class` from rule content (project-specific details → `project-pattern`, universal → `general`)
5. Check deduplication: if `source_comment_id` already exists in either DB → output `SKIPPED <id> already_exists`, continue
6. Generate ID, insert, output `SAVED <id>`

**Python import:**
```python
from lib.corrections import load_corrections, add_correction, update_status, parse_and_save
```

## `@cao-learn` Detection in Agents

### Trust model

`@cao-learn` is a **trusted user instruction**, not ticket content. It is explicitly exempt from prompt-injection-guard because:
- It uses a specific structured tag (`@cao-learn`) with defined fields
- It only triggers a DB write (no shell execution, no file modification, no label change)
- The agent validates the format before acting — malformed `@cao-learn` blocks are ignored
- The `promote` command (which modifies files) is a separate, explicit user action via CLI — not triggered by ticket content

The injection guard is updated to document this exception precisely: only `@cao-learn` with valid `rule` field is trusted. Any `@cao-learn` block containing injection red flags (shell commands, `ignore previous instructions`, role overrides) is treated as a standard injection attempt and logged.

### Detection logic in agent step 1 (Load context)

```bash
LEARN_COMMENTS=$(gh issue view "$TICKET_N" --repo "$OWNER/$REPO" \
  --json comments \
  --jq '[.comments[] | select(.body | contains("@cao-learn"))]')

if [ -n "$LEARN_COMMENTS" ] && [ "$LEARN_COMMENTS" != "[]" ]; then
  python3 lib/corrections.py parse-and-save \
    --comments "$LEARN_COMMENTS" \
    --agent "$AGENT_NAME" \
    --source "$OWNER/$REPO#$TICKET_N" \
    --project-db "$HOME/.claude/projects/${PROJECT_SLUG}/cao.db" \
    --global-db  "$HOME/.claude/cao.db"
fi
```

## Migration from step 5.1 markdown files

The previous meta-learning implementation (step 5.1 in `chief-builder/agent.md`) wrote to:
- `~/.claude/projects/<slug>/memory/enrichment-patterns.md`
- `~/.claude/memory/chief-builder-methodology.md`

**Migration strategy**: these files are not deleted and continue to be loaded at agent startup (step 1 "Load accumulated learnings" block) until they are explicitly migrated. The user can migrate manually:

```bash
/cao-corrections add --agent chief-builder --rule "..." --gap "..." --class project-pattern
```

Once migrated, the user removes the source markdown file. Step 5.1 is removed from `agent.md` — the DB system replaces it entirely.

## Out of Scope

- UI for browsing corrections (CLI only for now)
- Automatic correction extraction without `@cao-learn` (user must opt in)
- Syncing corrections across machines
- Corrections versioning / rollback
- Migrating existing markdown learning files automatically
